## 🚀 Backlog фич

| Приоритет | Фича | Описание |
|-----------|------|----------|
| 🔴 High | Персистентный message store | In-memory store теряется при перезапуске — reply/edit/delete перестают работать. Нужна простая SQLite/JSON персистентность с TTL |
| 🔴 High | Асимметричные пользователи | Поддержка конфигурации, где у пользователя есть аккаунт только в одном мессенджере (только TG или только MAX). Сейчас оба поля обязательны. Нужна опциональность `telegram_user_id` / `max_user_id` и соответствующая логика sender matching |
| 🔴 High | Упрощённый процесс подключения MAX | Текущий flow (diagnose.py → auth.py → ввод телефона/кода) сложен. Исследовать и реализовать более простой способ — возможно через QR-код или веб-токен без SMS. Также рассмотреть: extract_max_token.js как основной (менее трудоёмкий) путь |
| 🔴 High | Утилита динамического конфигурирования | CLI или интерактивный скрипт для управления конфигом без ручного редактирования YAML: добавление/удаление чатов и пользователей, авто-определение ID чатов (по ссылке/названию), авто-определение user ID, валидация и перезапуск бриджа |
| 🔴 High | Исправить delete в обе стороны | TG→MAX: Pyrogram не сообщает `chat_id` в обычных группах. MAX→TG: MAX WS не шлёт опкод 66 для собственных сообщений. Исследовать обходные пути (опрос истории, отдельный API запрос) |
| 🔴 High | Telegram-бот управления мостом | Бот для мониторинга и управления бриджем через Telegram: статус, перезапуск, логи, список активных чатов. Исследовать реализацию на базе OpenClaw (агентный фреймворк) |
| 🔴 High | Агент удалённого конфигурирования | Telegram-агент (на базе OpenClaw) для динамического управления конфигом прямо через чат: добавление/удаление чатов и пользователей, авто-определение ID по ссылке или названию, запуск авторизации, применение изменений без перезапуска. Требует исследования возможностей OpenClaw |
| 🟡 Medium | Исследование OpenClaw | Изучить OpenClaw как фреймворк для Telegram-ботов/агентов: возможности, ограничения, подходит ли для управляющего бота и агента конфигурирования. Оценить альтернативы (aiogram, python-telegram-bot) |
| 🟡 Medium | Graceful reconnect MAX WebSocket | При разрыве соединения reconnect работает, но store и tracker в памяти сохраняются — проверить сценарий длительного отключения |
| 🟡 Medium | Форматирование текста (bold/italic/code) | Конвертация Markdown (TG) ↔ MAX разметки |
| 🟡 Medium | Пересылка нескольких медиафайлов в одном сообщении | Сейчас обрабатывается только первый аттач в сообщении |
| 🟢 Low | Мониторинг / healthcheck | HTTP endpoint `/health` для Docker healthcheck |
| 🟢 Low | Метрики | Счётчики переданных сообщений по направлениям |
| 🟢 Low | Поддержка реакций | Синхронизация emoji-реакций между мессенджерами |
| 🟢 Low | Поддержка опросов (polls) | Пересылка как текст с вариантами ответов |

## 🐛 Известные ограничения (не баги, а по-другому нельзя)

| Ограничение | Причина |
|-------------|---------|
| Delete TG→MAX не работает в обычных TG-группах | Pyrogram не сообщает `chat_id` для DeletedMessages в обычных группах (только в супергруппах/каналах) |
| Delete MAX→TG не работает для сообщений, отправленных аккаунтом бриджа | MAX WS не присылает опкод 66 для сообщений, отправленных тем же аккаунтом |
| Reply/edit/delete теряются после перезапуска | In-memory store очищается — нет персистентности |
| Форматирование не сохраняется | Разные форматы разметки в TG и MAX |
