# Telegram ↔ MAX Bridge

Двунаправленный мост между Telegram и MAX (бывший VK Teams / MyTeam). Сообщения автоматически зеркалируются из одного мессенджера в другой и обратно.

## Описание

### Зачем нужны user accounts, а не боты

Telegram-бот не может читать обычные сообщения в группах, куда его не добавили явно, и не имеет доступа к истории. Бот MAX тоже ограничен. Вместо этого мост использует **реальные пользовательские аккаунты** через MTProto (Pyrogram) для Telegram и WebSocket API (vkmax) для MAX.

Это позволяет:
- читать все сообщения в группе без специальных разрешений администратора;
- отправлять сообщения от имени конкретного пользователя, а не безликого бота — в чатах будет видно настоящее имя.

### Схема работы

```
Telegram-группа                    MAX-чат
     │                                 │
     │  новое сообщение от Alice        │
     ▼                                 │
[TG Listener-аккаунт]                 │
     │  читает сообщение               │
     ▼                                 │
  [Bridge]  ──────────────────►  [MAX-аккаунт Alice]
                                       │  отправляет зеркало
                                       ▼
                               MAX-чат (от имени Alice)
```

- **Listener-аккаунт** — один на каждый мессенджер, слушает входящие сообщения.
- **User-аккаунты** — по одному на каждого пользователя в каждом мессенджере, отправляют зеркальные сообщения.
- Если отправитель не найден в таблице users, мост всё равно пересылает сообщение через listener-аккаунт с указанием имени в тексте.

Поддерживаются: текст, фото, видео, файлы, аудио, ответы на сообщения, редактирование и удаление сообщений.

## Шаг 1: Получение Telegram API credentials

1. Откройте [my.telegram.org](https://my.telegram.org) и войдите в свой аккаунт.
2. Перейдите в раздел **API development tools**.
3. Создайте приложение (название и описание — произвольные).
4. Скопируйте **App api_id** (число) и **App api_hash** (строка из 32 символов).

Эти данные вставляются в `config.yaml` как `api_id` и `api_hash`.

> Важно: `api_id` и `api_hash` — общие для **всех** Telegram-аккаунтов, задействованных в мосту. Создавать отдельные приложения на my.telegram.org для каждого пользователя не нужно.

## Шаг 2: Заполнение config.yaml

Скопируйте пример конфига:

```bash
cp config.example.yaml config.yaml
```

Затем отредактируйте `config.yaml`. Ниже — описание каждого поля.

### Полный пример

```yaml
api_id: 12345678
api_hash: "0123456789abcdef0123456789abcdef"

listener_telegram_session: "tg_listener"
listener_max_session: "max_listener"

sessions_dir: "sessions"

chat_pairs:
  - name: "team-general"
    telegram_chat_id: -1001234567890
    max_chat_id: -68093732121255

users:
  - name: "Alice"
    telegram_user_id: 111111111
    max_user_id: 175123456789
    telegram_session: "tg_alice"
    max_session: "max_alice"

  - name: "Bob"
    telegram_user_id: 333333333
    max_user_id: 175987654321
    telegram_session: "tg_bob"
    max_session: "max_bob"
```

### Описание полей

#### `api_id` и `api_hash`

Получены на [my.telegram.org](https://my.telegram.org) (см. Шаг 1).

#### `listener_telegram_session` и `listener_max_session`

Произвольное имя файла сессии для listener-аккаунта. Файл будет создан в директории `sessions/` при авторизации.

- `listener_telegram_session: "tg_listener"` → файл `sessions/tg_listener.session`
- `listener_max_session: "max_listener"` → файл `sessions/max_listener.max_session`

Listener-аккаунт должен быть **участником всех чатов**, перечисленных в `chat_pairs`.

#### `sessions_dir`

Директория для хранения файлов сессий (по умолчанию `sessions`). При запуске через Docker монтируется как volume.

#### `chat_pairs`

Список пар чатов для зеркалирования.

| Поле | Описание |
|------|----------|
| `name` | Произвольное название пары (используется в логах) |
| `telegram_chat_id` | Числовой ID чата/группы/канала в Telegram |
| `max_chat_id` | Числовой ID чата в MAX |

**Как узнать `telegram_chat_id`:**

- **Способ 1 (рекомендуется):** Перешлите любое сообщение из нужной группы боту [@userinfobot](https://t.me/userinfobot). Он ответит с ID чата.
- **Способ 2:** В Telegram Desktop включите экспериментальные функции: `Settings → Advanced → Experimental features → Show Peer IDs`. ID отобразится прямо в заголовке чата.
- Для супергрупп и каналов ID начинается с `-100`.

**Как узнать `max_chat_id`:**

- Откройте нужный чат в MAX (веб-версия или десктоп-приложение).
- ID чата виден в URL страницы чата: `.../@chat/−68093732121255`.
- Альтернативно — используйте MAX API или скрипт аутентификации.

#### `users`

Список пользователей, чьи аккаунты подключены к мосту (максимум 10).

| Поле | Описание |
|------|----------|
| `name` | Отображаемое имя (используется в логах) |
| `telegram_user_id` | Числовой ID пользователя в Telegram |
| `max_user_id` | Числовой ID пользователя в MAX |
| `telegram_session` | Имя файла сессии Telegram для этого пользователя |
| `max_session` | Имя файла сессии MAX для этого пользователя |

**Как узнать `telegram_user_id`:**

Напишите боту [@userinfobot](https://t.me/userinfobot) — он ответит вашим числовым ID (например `111111111`).

**Как узнать `max_user_id`:**

- Откройте профиль пользователя в MAX.
- ID виден в URL профиля.
- Или используйте скрипт авторизации: после входа в аккаунт ID выводится в логах MAX API.

## Шаг 3: Авторизация аккаунтов

Перед первым запуском необходимо авторизовать все аккаунты:

```bash
python -m src.auth
```

Скрипт поочерёдно авторизует:
1. Listener-аккаунт Telegram
2. Listener-аккаунт MAX
3. Каждого пользователя из `users` (Telegram + MAX)

**Для Telegram:** Pyrogram запросит номер телефона и код из SMS/Telegram. Введите данные аккаунта, от имени которого будет работать listener или данный пользователь.

**Для MAX:** Скрипт запросит номер телефона. На него придёт SMS с кодом, который нужно ввести в терминал.

После успешной авторизации файлы сессий сохраняются в директории `sessions/`:
- Telegram-сессии: `sessions/<имя>.session` (бинарный формат Pyrogram)
- MAX-сессии: `sessions/<имя>.max_session` (JSON с токеном)

Если сессия уже существует, скрипт проверяет её валидность и пропускает повторную авторизацию. Если MAX-сессия истекла, скрипт предложит пройти авторизацию заново.

> Файлы сессий содержат токены доступа к аккаунтам. Не публикуйте их и не передавайте третьим лицам.

## Шаг 4: Запуск

### Напрямую (Python)

```bash
python -m src
```

### Docker Compose

```bash
# Собрать образ и запустить
docker compose up -d

# Просмотр логов
docker compose logs -f

# Остановка
docker compose down
```

В `docker-compose.yml` уже настроены:
- монтирование `config.yaml` (read-only)
- монтирование директории `sessions/` (read-write для хранения сессий)
- политика перезапуска `unless-stopped`

> При запуске через Docker авторизацию (`python -m src.auth`) необходимо выполнить **на хосте** (или в отдельном контейнере с интерактивным терминалом), так как ввод кода из SMS требует TTY.

## Архитектура

```
src/
├── main.py              # Точка входа, инициализация компонентов
├── config.py            # Загрузка и валидация config.yaml
├── types.py             # Датаклассы: AppConfig, ChatPair, UserMapping, BridgeEvent
├── auth.py              # Интерактивная авторизация аккаунтов
├── message_store.py     # In-memory хранилище маппинга ID сообщений (TTL 24h)
├── bridge/
│   ├── bridge.py        # Основная логика моста, роутинг событий
│   └── echo_guard.py    # Защита от дублирования (echo prevention)
├── telegram/
│   ├── listener.py      # Pyrogram-клиент, слушает входящие сообщения TG
│   └── client_pool.py   # Пул Pyrogram-клиентов для user-аккаунтов TG
└── max/
    ├── listener.py      # vkmax-клиент, слушает WebSocket MAX
    ├── client_pool.py   # Пул MAX-клиентов для user-аккаунтов
    ├── session.py       # Сохранение/загрузка MAX-токенов
    └── media.py         # Скачивание медиафайлов из MAX
```

### Поток данных

1. **TelegramListener** или **MaxListener** получает событие (новое сообщение, редактирование, удаление).
2. Listener создаёт `BridgeEvent` и передаёт его в **Bridge**.
3. **EchoGuard** проверяет, что сообщение не от управляемого аккаунта (предотвращает бесконечный цикл).
4. **Bridge** находит нужную пару чатов через `ConfigLookup` и определяет, от какого аккаунта отправить зеркало.
5. Если пользователь найден в `users` — сообщение отправляется от его аккаунта. Иначе — через listener с подписью `[Имя]: текст`.
6. **MessageStore** хранит соответствие ID оригинального сообщения и зеркала (для корректной обработки редактирований, удалений и ответов).

## Ограничения

| Функция | Статус |
|---------|--------|
| Текстовые сообщения | Поддерживается |
| Фото | Поддерживается |
| Видео | Поддерживается |
| Файлы/документы | Поддерживается |
| Аудиофайлы | Поддерживается |
| Голосовые сообщения | Частично: передаются как аудиофайл `.ogg`, воспроизведение как "голосовое" может не работать |
| Стикеры | Не поддерживается: отображается текст `[Sticker]` |
| Редактирование сообщений | Поддерживается |
| Удаление сообщений | В Telegram — только для каналов (API не предоставляет chat_id для удалений в группах). В MAX — поддерживается |
| Ответы на сообщения (reply) | Поддерживается |
| Реакции | Не поддерживается |
| Опросы / голосования | Не поддерживается |
| Форматирование Markdown | Не сохраняется при пересылке |

## Troubleshooting

### `RuntimeError: MAX listener session not found`

Сессия не создана. Запустите авторизацию:

```bash
python -m src.auth
```

### `MAX session expired. Re-authenticating...`

Токен MAX-аккаунта истёк (обычно через несколько недель неактивности). Удалите файл сессии и повторно авторизуйтесь:

```bash
rm sessions/<имя>.max_session
python -m src.auth
```

### `Telegram session not found` / `AuthKeyUnregistered`

Pyrogram-сессия недействительна или удалена другим устройством. Удалите файл сессии и повторно авторизуйтесь:

```bash
rm sessions/<имя>.session
python -m src.auth
```

### `chat_pairs[0].telegram_chat_id is required` / чат не найден

Проверьте правильность ID чата в `config.yaml`. Listener-аккаунт должен быть **участником** этого чата.

### Сообщения дублируются

Убедитесь, что listener-аккаунт и user-аккаунты — это **разные** Telegram/MAX аккаунты. Если один аккаунт используется и как listener, и как user, EchoGuard предотвратит обработку его сообщений.

### Сообщения не пересылаются в одну сторону

- Проверьте, что listener-аккаунт в нужном мессенджере является участником чата.
- Проверьте логи на ошибки подключения.
- Для MAX: WebSocket может переподключаться после разрыва — это нормально, reconnect происходит автоматически.
